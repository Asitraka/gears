GEARS
Geant4 Example Application with Rich features and Small footprints
README.md
1 [GEARS][] is a [Geant4][] Example Application with Rich features yet Small footprint. The entire C++ coding is minimized down to a single file with less than 500 [SLOC][]. This is achieved mainly by utilizing [Geant4][] plain [text geometry description][tg], [build-in UI commands][TUI] (macros), and C++ inheritance. It is ideal for student training and fast implementation of small to medium-sized experiments.
2 
3 # Features
4 
5 * Single small C++ file

7  * Easy management and compiling
8 * Fast compilation
9  * a few second on a regular PC
10 * [Output](#Output) in ROOT TTree format
11  * Build-in data compression
12  * Fast access to independent data members
13  * Flat tree (no nested branches or arrays)
14  * Easy to use in TTree::Draw
15  * No need to load extra library to open
16 * [Record information of step 0](#Record-information-of-step-0) (initStep)
17  * This is not available from G4UserSteppingAction
18 * Create/Change geometry without re-compilation
19 * Turn any volume to a sensitive detector by adding "(S)" in its name
20 * Assign optical properties in [Geant4][] plain [text geometry description][tg]
21 * Optional optical and hadronic physics
22 * Periodic status report in a long run
23 * Frequently used source spectra (AmBe, Am-241, etc.)
24 * [Doxygen documentation](http://www.physino.xyz/gears/)

43 ## Geometry
44 [Geant4][] provides a simple way to describe the geometry of a detector in a plain text file. For example, the following line in such a file defines an experimental hall filled with air, the dimension of which is 20 x 20 x 30 meters:
45 
46 ~~~
47 :volume hall BOX 10*m 10*m 10*m G4_AIR
48 ~~~
49 
50 More examples can be found in the [geom/](geom/) directory, such as [geom/hall.tg](geom/hall.tg). Files in this directory have a suffix of *.tg*, indicating that they are [text geometry][tg] description files. A [Geant4][] macro command is added to load [one of the geometry files](geom/Rutherford/foil.tg):
51 
52 ~~~
53 /geometry/source geom/Rutherford/foil.tg
54 ~~~
55 
56 The command must be used before [/run/initialize][run], otherwise [Geant4][] does not have a detector to initialize.

58 ## Material
59 
60 The [NIST][] material table provided by Geant4 contains a lot of commonly used materials. One can run /material/nist/[listMaterials][] at any Geant4 state to print the list locally. These materials can be used directly in a [text geometry description][tg], for example
61 
62 ~~~
63 // use NIST material G4_AIR to define vacuum
64 :mixt vacuum 1e-9 1 G4_AIR 1
65 ~~~

67 ## Optical properties of a material or a surface
68 
69 There is no tag to define the optical properties of a material or a surface in the default [Geant4][] [text geometry description][tg]. The following two tags are added in [GEARS][] to enable definition of optical materials and surfaces using [Geant4][] [text geometry description][tg] syntax:
70 
71 ### Define optical properties of a material
72 
73 ~~~
74 //:prop <material>
75 // <wavelength-independent_property> <value>
76 :prop pureCsIat77K
77  SCINTILLATIONYIELD 100./keV
78  RESOLUTIONSCALE 2.
79  FASTTIMECONSTANT 1.*ns
80  SLOWTIMECONSTANT 1.*us
81  YIELDRATIO 0.8
82 
83 //:prop <material> photon_energies <int(array size)> <energy array>
84 // <wavelength-dependent_property> <property values>
85 :prop pureCsIat77K
86  photon_energies 2 2.034*eV, 3.025*eV, 4.136*eV
87  RINDEX 1.34, 1.35, 1.36
88  ABSLENGTH 1.0*meter, 1.1*meter, 1.2*meter
89 ~~~
90 
91 ### Define optical properties of a surface
92 
93 First of all, there is no need to define a *surface* for polished interfaces between two media. As long as the two media have an index of refraction stored in their respective G4MaterialPropertiesTable, the [G4OpBoundaryProcess][]::[PostStepDoIt][] will handle the refraction and reflection correctly.
94 
95 One can use the following syntax to define a [G4LogicalBorderSurface][] in case that there is a real need to specify the optical properties of the interface:
96 
97 ~~~
98 //:surf v12v2 v1:copyNo1 v2:copyNo2
99 :surf CsI2Teflon CsI:1 Teflon:1
100  type dielectric_dielectric
101  model unified
102  finish ground
103  sigma_alpha 0.1
104  property photon_energies 2 2.5*eV 5.0*eV
105  REFLECTIVITY 0.9 0.9
106 //property must be the last setting due to the current coding method
107 ~~~
108 
109 Note that physics volumes from the same logical volume created by the text geometry processor share the same name as their logical volume. Since [G4LogicalBorderSurface][] requires pointers to the two physical volumes beside, a unique copy number has to be attached to the volume name to uniquely identify the physics volume.

111 ## Sensitive detector
112 
113 Sensitive detectors are specified by simply add "(S)" at the end of their volume names. The copy numbers of their volumes must be continuous integers starting from 0.

115 # Physics
116 
117 ## Physics list
118 
119 Since it requires a lot of knowledge to set up a correct physics list, [Geant4][] provides some pre-packaged physics lists. Three are enabled by default in [GEARS][]:
120 
121 - G4DecayPhysics, it defines particle decays. It defines more particles than G4EmStandardPhysics does, so it has to be enabled before optional hadronic physics, otherwise the later will complain about missing particle definitions.
122 - G4RadioactiveDecayPhysics, it defines nuclear decays.
123 - G4EmStandardPhysics, it has to be defined after G4RadioactiveDecayPhysics, otherwise nuclei cannot get their energy state information and won't decay. No [Geant4][] documentation warns it.
124 
125 Three optional lists (G4OpticalPhysics, G4HadronElasticPhysicsHP, G4HadronPhysicsFTFP_BERT_HP) can be enabled using [Geant4][] macros:
126 
127 ~~~
128  # has to be called before /run/initialize
129  /physics_lists/enable Optical
130  /physics_lists/enable HadronElastic
131  /physics_lists/enable HadronInelastic
132 ~~~
133

135 
136 Run `/process/list` after `/run/initialize`, and you will get

139  Transportation, Decay, RadioactiveDecay, msc
140  hIoni, ionIoni, hBrems, hPairProd
141  CoulombScat, eIoni, eBrem, annihil
142  phot, compt, conv, muIoni
143  muBrems, muPairProd, hadElastic, neutronInelastic
144  nCapture, nFission, protonInelastic, pi+Inelastic
145  pi-Inelastic, kaon+Inelastic, kaon-Inelastic, kaon0LInelastic
146  kaon0SInelastic, lambdaInelastic,anti-lambdaInelastic, sigma-Inelastic
147 anti_sigma-Inelastic, sigma+Inelastic,anti_sigma+Inelastic, xi-Inelastic
148  anti_xi-Inelastic, xi0Inelastic, anti_xi0Inelastic, omega-Inelastic
149 anti_omega-Inelastic,anti_protonInelastic,anti_neutronInelastic,anti_deuteronInelastic
150 anti_tritonInelastic, anti_He3Inelastic,anti_alphaInelastic
151 ~~~
152 
153 Now you can use, for example, `/process/inactivate nCapture` to disable neutron capture process in your simulation. And you can use, for example, `/process/setVerbose 20 RadioactiveDecay` to change the verbosity of the radioactive decay process.
154 
155 Individual optical processes can be toggled by the following commands:
156 
157 ~~~
158 /process/optical/processActivation Cerenkov true/false
159 /process/optical/processActivation Scintillation true/false
160 /process/optical/processActivation OpAbsorption true/false
161 /process/optical/processActivation OpRayleigh true/false
162 /process/optical/processActivation OpMieHG true/false
163 /process/optical/processActivation OpBoundary true/false
164 /process/optical/processActivation OpWLS true/false
165 ~~~
166 
167 Detailed control of radioactive decay is provided by the /[grdm]/ command, for example,
168 
169 ~~~
170 /grdm/deselectVolume chamber # disabled radioactive decay in volume "chamber"
171 /grdm/nucleusLimits 1 80 # enabled radioactive decay only when z in [1, 80]
172 ~~~

174 # Output
175 
176 ## Track point
177 
178 A track point is a concept introduced in [GEARS][]. It is a point where a track is generated or changed. It records the following information:
179 
180 * Track id
181 * Step number, starting from 0
182 * Detector volume copy number
183 * [Process id](#process-id)
184 * [Particle id](#particle-id)
185 * Parent id
186 * Energy deposited [keV]
187 * Kinetic energy of the track [keV]
188 * global time [ns]
189 * x [mm]
190 * y [mm]
191 * z [mm]
192 
193 ## Process id
194 
195 The physics process generating each track point is saved in a variable `pro[n]`, where `n` is the index of the track point. It equals to (process type)*1000 + (sub type). The Process types are defined in G4ProcessType.hh, sub types are defined in G4HadronicProcessType.hh, G4DecayProcessType.hh, G4EmProcessSubType.hh, G4TransportationProcessType.hh, G4FastSimulationProcessType.hh, G4OpProcessSubType.hh, etc. They can be found at http://www-geant4.kek.jp/lxr/find?string=Type.hh
196 
197 ## Particle id
198 
199 The type of particle related to a track point is saved in a variable `pdg[n]`. It is the same as the `PDG encoding` of the particle. A Google search will give more information about it.

201 ## Record information of step 0
202 
203 One cannot get access to step 0 (initStep printed on screen if `/tracking/verbose` is set to be more than 0) through [G4UserSteppingAction][], which only provides access to step 1 and afterwards. However, step 0 contains some very important information that is constantly requested by the user. For example, the energy of a gamma ray from a radioactive decay is only available from step 0. Such information can be easily displayed using the following ROOT command with the Output ROOT tree, `t`:
204 
205 ~~~
206  // draw kinetic energy, "k", of a gamma (pdg==22)
207  // created by radioactive decay process (pro==6210)
208  t->Draw("k","pro==6210 && pdg==22")
209 ~~~
210 
211 This is achieved by using [G4SteppingVerbose][] instead of [G4UserSteppingAction][] for data recording. The former provides a function called TrackingStarted() to print verbose information about step 0 on screen if `/tracking/verbose` is set to be more than 0. It also provides a function called StepInfo() to print verbose information about steps after step 0 on screen if `/tracking/verbose` is more than 0. [G4SteppingVerbose][]::StepInfo() is called right before [G4UserSteppingAction][]::UserSteppingAction(G4Step*) is called in [G4SteppingManager][]::Stepping(), it hence can be used to replace the functionality of [G4UserSteppingAction][]::UserSteppingAction(G4Step*).
212 
213 In fact, [G4UserSteppingAction][]::UserSteppingAction(G4Step*) is not used at all in [GEARS][]. The Output class inherits TrackingStarted() and StepInfo() from [G4SteppingVerbose][] to record data from all steps. There is another advantage of using [G4SteppingVerbose][] instead of [G4UserSteppingAction][] for recording, that is, [G4SteppingVerbose][] is provided as a globally available singleton, which can be easily accessed at different places in the codes using:
214 
215 ~~~
216  G4VSteppingVerbose::GetInstance()
217 ~~~
218 
219 This is used in [G4UserRunAction][] to open and close a TFile, in [G4UserEventAction][] to fill a TTree.
220 
221 The catch is that functions in [G4SteppingVerbose][] will not be called in [G4SteppingManager][] unless `/tracking/verbose` is set, which will print too much information on screen for a long run. This is solved in EventAction::BeginOfEventAction by turning on tracking verbose all the time so that all functions in [G4SteppingVerbose][] will be called, while at the same time, turning on [G4SteppingVerbose][] local verbose flag [Silent][] to run them in silent mode.
