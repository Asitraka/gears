GEARS
Geant4 Example Application with Rich features and Small footprints
README.md
Go to the documentation of this file.
1 [GEARS][] is a [Geant4][] Example Application with Rich features yet Small footprint. The entire C++ coding is minimized down to a single file with about 600 [SLOC][]. This is achieved mainly by utilizing [Geant4][] plain [text geometry description][tg], [build-in UI commands][TUI] (macros), and C++ inheritance. It is ideal for student training and fast implementation of small to medium-sized experiments.
2 
3 # Features
4 
5 * Single small C++ file
6  * Easy code browsing
7  * Easy management and compiling
8 * Fast compilation
9  * a few second on a regular PC
10 * [Output in JSON data format](#json) (default)
11  * Universal data format, easy to read by different tools
12  * Human readable ASCII file
13  * Capable of dealing with multiple dimensional arrays
14 * [Output](#output) in [ROOT](#root) TTree format (if ROOT is installed)
15  * Build-in data compression, well suitable for large data processing
16  * Fast access to independent data members
17  * Flat tree (no nested branches or arrays) with short leave names
18  * Easy to use in TTree::Draw
19  * No need to load extra library to open
20 * [Record information of step 0](#record-information-of-step-0) (initStep)
21  * This is not available from G4UserSteppingAction
22 * [simple text][tg] or [GDML][] geometry I/O
23  * Fast implementation of [detector geometry](#detector) without C++ programming
24  * Create/Change geometry without re-compilation
25  * Turn any volume to a sensitive detector by adding "(S)" in its name
26  * Assign optical properties in [Geant4][] plain [text geometry description][tg]
27 * Optional optical and hadronic physics
28 * Periodic status report in a long run
29 * Frequently used source spectra (AmBe, Am-241, etc.)
30 * [Doxygen documentation](http://www.physino.xyz/gears/)
31 * Many sample macros and geometry descriptions for feature demonstration

50 ## Geometry
51 [GEARS][] accepts two types of detector geometry descriptions in pure ASCII format as input:
52 - [Geant4][] [text geometry description][tg] (recommended)
53 - [GDML][] (provided for data analysis and visualization in other tools)
54 Their difference is similar to that between [markdown][md] and [HTML][]. The simpler [text geometry description][tg] provided by [Geant4][] is recommended to be used as [GEARS][]'s input given its simplicity and readability. For example, an experimental hall filled with air and of a dimension of 10 x 10 x 10 meters can be easily implemented using the following line:
55 
56 ~~~
57 :volume hall BOX 5*m 5*m 5*m G4_AIR
58 ~~~
59 
60 More examples can be found in the [geom/](geom/) directory, such as [geom/hall.tg](geom/hall.tg). Files in this directory have a suffix of *.tg*, indicating that they are [text geometry][tg] description files. A [Geant4][] macro command is added to load [one of the geometry files](examples/Rutherford/foil.tg):
61 
62 ~~~
63 /geometry/source examples/Rutherford/foil.tg
64 ~~~
65 
66 Alternatively,
67 
68 ~~~
69 /geometry/source file.gdml
70 ~~~
71 
72 The command must be used before [/run/initialize][run], otherwise [GEARS][] will construct a simple experimental hall automatically to prevent crashing.
73 
74 [GEARS][] provides the following command to export constructed detector geometry to a [GDML][] file:
75 
76 ~~~
77 /geometry/export output.gdml
78 ~~~
79 
80 This can only be used after the macro command [/run/initialize][run], which constructs the detector geometry before exporting. While the simpler [text geometry description][tg] can only be understood by [Geant4][], [GDML][] can be understood by many other tools. For example, [ROOT][] provides functions to read and visualize [GDML][] geometries. On the other hand, it is not that easy to write a valid [GDML][] file manually. This functionality is provided to enable the following usage:
81 
82 ~~~
83 # describe the detector using simple text geometry description
84 /geometry/source input.tg
85 # construct the detector
86 /run/intialize
87 # export detector geometry as GDML for analysis/visualization in other tools
88 /geometry/export output.gdml
89 ~~~

91 ## Material
92 
93 The [NIST][] material table provided by Geant4 contains all elements (C, H, O, for example) and a lot of commonly used materials (start with "G4_"). One can run /material/nist/[listMaterials][] at any Geant4 state to print the list locally. These materials can be used directly in a [text geometry description][tg], for example
94 
95 ~~~
96 // use Geant4 elements, C and H to define TPB
97 :MIXT_BY_NATOMS TPB 1.079 2 C 28 H 22
98 // use NIST material G4_AIR to define vacuum
99 :mixt vacuum 1e-9 1 G4_AIR 1
100 ~~~

102 ## Optical properties of a material or a surface
103 
104 There is no tag to define the optical properties of a material or a surface in the default [Geant4][] [text geometry description][tg]. The following two tags are added in [GEARS][] to enable definition of optical materials and surfaces using [Geant4][] [text geometry description][tg] syntax:
105 
106 ### Define optical properties of a material
107 
108 ~~~
109 //:prop <material>
110 // <wavelength-independent_property> <value>
111 :prop pureCsIat77K
112  SCINTILLATIONYIELD 100./keV
113  RESOLUTIONSCALE 2.
114  FASTTIMECONSTANT 1.*ns
115  SLOWTIMECONSTANT 1.*us
116  YIELDRATIO 0.8
117 
118 //:prop <material> photon_energies <int(array size)> <energy array>
119 // <wavelength-dependent_property> <property values>
120 :prop pureCsIat77K
121  photon_energies 2 2.034*eV, 3.025*eV, 4.136*eV
122  RINDEX 1.34, 1.35, 1.36
123  ABSLENGTH 1.0*meter, 1.1*meter, 1.2*meter
124 ~~~
125 
126 ### Define optical properties of a surface
127 
128 First of all, there is no need to define a *surface* for polished interfaces between two media. As long as the two media have an index of refraction stored in their respective G4MaterialPropertiesTable, the [G4OpBoundaryProcess][]::[PostStepDoIt][] will handle the refraction and reflection correctly.
129 
130 One can use the following syntax to define a [G4LogicalBorderSurface][] in case that there is a real need to specify the optical properties of the interface:
131 
132 ~~~
133 //:surf v12v2 v1:copyNo1 v2:copyNo2
134 :surf CsI2Teflon CsI:1 Teflon:1
135  type dielectric_dielectric
136  model unified
137  finish ground
138  sigma_alpha 0.1
139  property photon_energies 2 2.5*eV 5.0*eV
140  REFLECTIVITY 0.9 0.9
141 //property must be the last setting due to the current coding method
142 ~~~
143 
144 Note that physics volumes from the same logical volume created by the text geometry processor share the same name as their logical volume. Since [G4LogicalBorderSurface][] requires pointers to the two physical volumes beside, a unique copy number has to be attached to the volume name to uniquely identify the physics volume.

147 
148 Sensitive detectors are specified by simply add "(S)" at the end of their volume names. The copy numbers of their volumes must be continuous integers starting from 0.

150 # Physics
151 
152 ## Physics list
153 
154 Since it requires a lot of knowledge to set up a correct physics list, [Geant4][] provides some pre-packaged physics lists. Three are enabled by default in [GEARS][]:
155 
156 - G4DecayPhysics, it defines particle decays. It defines more particles than G4EmStandardPhysics does, so it has to be enabled before optional hadronic physics, otherwise the later will complain about missing particle definitions.
157 - G4RadioactiveDecayPhysics, it defines nuclear decays.
158 - G4EmStandardPhysics, it has to be defined after G4RadioactiveDecayPhysics, otherwise nuclei cannot get their energy state information and won't decay. No [Geant4][] documentation warns it.
159 
160 Three optional lists (G4OpticalPhysics, G4HadronElasticPhysicsHP, G4HadronPhysicsFTFP_BERT_HP) can be enabled using [Geant4][] macros:
161 
162 ~~~
163  # has to be called before /run/initialize
164  /physics_lists/enable Optical
165  /physics_lists/enable HadronElastic
166  /physics_lists/enable HadronInelastic
167 ~~~
168 
169 ### Physics processes
170 
171 Run `/process/list` after `/run/initialize`, and you will get
172 
173 ~~~
174  Transportation, Decay, RadioactiveDecay, msc
175  hIoni, ionIoni, hBrems, hPairProd
176  CoulombScat, eIoni, eBrem, annihil
177  phot, compt, conv, muIoni
178  muBrems, muPairProd, hadElastic, neutronInelastic
179  nCapture, nFission, protonInelastic, pi+Inelastic
180  pi-Inelastic, kaon+Inelastic, kaon-Inelastic, kaon0LInelastic
181  kaon0SInelastic, lambdaInelastic,anti-lambdaInelastic, sigma-Inelastic
182 anti_sigma-Inelastic, sigma+Inelastic,anti_sigma+Inelastic, xi-Inelastic
183  anti_xi-Inelastic, xi0Inelastic, anti_xi0Inelastic, omega-Inelastic
184 anti_omega-Inelastic,anti_protonInelastic,anti_neutronInelastic,anti_deuteronInelastic
185 anti_tritonInelastic, anti_He3Inelastic,anti_alphaInelastic
186 ~~~
187 
188 Now you can use, for example, `/process/inactivate nCapture` to disable neutron capture process in your simulation. And you can use, for example, `/process/setVerbose 20 RadioactiveDecay` to change the verbosity of the radioactive decay process.
189 
190 Individual optical processes can be toggled by the following commands:
191 
192 ~~~
193 /process/optical/processActivation Cerenkov true/false
194 /process/optical/processActivation Scintillation true/false
195 /process/optical/processActivation OpAbsorption true/false
196 /process/optical/processActivation OpRayleigh true/false
197 /process/optical/processActivation OpMieHG true/false
198 /process/optical/processActivation OpBoundary true/false
199 /process/optical/processActivation OpWLS true/false
200 ~~~
201 
202 Detailed control of radioactive decay is provided by the /[grdm]/ command, for example,
203 
204 ~~~
205 /grdm/deselectVolume chamber # disabled radioactive decay in volume "chamber"
206 /grdm/nucleusLimits 1 80 # enabled radioactive decay only when z in [1, 80]
207 ~~~

209 # Output
210 
211 ## Track point
212 
213 A track point is a concept introduced in [GEARS][]. It is a point where a track is generated or changed. It records the following information:
214 
215 * Track id (`trk` in short)
216 * Step number, starting from 0 (`stp` in short)
217 * Detector volume copy number (`det` in short)
218 * [Process id](#process-id) (`pro` in short)
219 * [Particle id](#particle-id) (`pid` in short)
220 * Parent id (`mom` in short)
221 * Energy deposited [keV] (`e` in short)
222 * Kinetic energy of the track [keV] (`k` in short)
223 * global time [ns] (`t` in short)
224 * x [mm]
225 * y [mm]
226 * z [mm]
227 
228 ## Process id
229 
230 The physics process generating each track point is saved in a variable `pro[n]`, where `n` is the index of the track point. It equals to (process type) * 1000 + (sub type). The Process types are defined in G4ProcessType.hh, sub types are defined in G4HadronicProcessType.hh, G4DecayProcessType.hh, G4EmProcessSubType.hh, G4TransportationProcessType.hh, G4FastSimulationProcessType.hh, G4OpProcessSubType.hh, etc. They can be found at http://www-geant4.kek.jp/lxr/find?string=Type.hh
231 
232 ## Particle id
233 
234 The type of particle related to a track point is saved in a variable `pdg[n]`. It is the same as the `PDG encoding` of the particle. A Google search will give more information about it.

236 ## Record information of step 0
237 
238 One cannot get access to step 0 (initStep printed on screen if `/tracking/verbose` is set to be more than 0) through [G4UserSteppingAction][], which only provides access to step 1 and afterwards. However, step 0 contains some very important information that is constantly requested by the user. For example, the energy of a gamma ray from a radioactive decay is only available from step 0. Such information can be easily displayed using the following ROOT command with the Output ROOT tree, `t`:
239 
240 ~~~cpp
241  // draw kinetic energy, "k", of a gamma (pdg==22)
242  // created by radioactive decay process (pro==6210)
243  t->Draw("k","pro==6210 && pdg==22")
244 ~~~
245 
246 This is achieved by using [G4SteppingVerbose][] instead of [G4UserSteppingAction][] for data recording. The former provides a function called [TrackingStarted][]() to print verbose information about step 0 on screen if `/tracking/verbose` is set to be more than 0. It also provides a function called [StepInfo][]() to print verbose information about steps after step 0 on screen if `/tracking/verbose` is more than 0. [G4SteppingVerbose][]::[StepInfo][]() is called right before [G4UserSteppingAction][]::[UserSteppingAction][]([G4Step][]*) is called in [G4SteppingManager][]::[Stepping][](), it hence can be used to fully replace the functionality of [G4UserSteppingAction][]::[UserSteppingAction][]([G4Step][]*).
247 
248 In fact, [G4UserSteppingAction][]::[UserSteppingAction][]([G4Step][]*) is not used at all in [GEARS][]. The Output class inherits [TrackingStarted][]() and [StepInfo][]() from [G4SteppingVerbose][] to record data from all steps. There is another advantage of using [G4SteppingVerbose][] instead of [G4UserSteppingAction][] for recording, that is, [G4SteppingVerbose][] is provided as a globally available singleton, which can be easily accessed at different places in the codes using:
249 
250 ~~~cpp
251  G4VSteppingVerbose::GetInstance()
252 ~~~
253 
254 This is used in [G4UserRunAction][] to open and close a TFile, in [G4UserEventAction][] to fill a TTree.
255 
256 The catch is that functions in [G4SteppingVerbose][] will not be called in [G4SteppingManager][] unless `/tracking/verbose` is set, which will print too much information on screen for a long run. This is solved in EventAction::BeginOfEventAction by turning on tracking verbose all the time so that all functions in [G4SteppingVerbose][] will be called, while at the same time, turning on [G4SteppingVerbose][] local verbose flag [Silent][] to run them in silent mode.

259 Gears provides two output formats: JSON and ROOT. The output file name and format can be chosen using macro command:

262 /run/output file.json (or file.root)
263 ~~~

267 ### JSON
268 Since the amount of data which gears might generate can be huge, we suggest install the [ROOT](#root) library and use .root format to store data to increase the preforance of data processing. If users do not want to use .root format, JSON format is avilable as default.
269 
270 All events are store as a JSON object array. Each event is an object and contains all information listed in [Track Point](#track-point), for example:
271 
272 ~~~
273 [
274 {
275  "n":2,
276  "trk":[1,1],
277  "stp":[0,1],
278  "det":[0,0],
279  "pro":[-1,1091],
280  "pdg":[22,22],
281  "mom":[0,0],
282  "e":[0,0],
283  "k":[4438.84,4438.84],
284  "t":[0,0.258763],
285  "x":[-1.99299,65.9771],
286  "y":[-8.12482,-45.244],
287  "z":[-225,-229.486]
288 },
289 {
290  "n":3,
291  "trk":[1,1,2],
292  "stp":[0,1,3],
293  "det":[0,0,2],
294  "pro":[-1,1091,2000],
295  "pdg":[22,22,22],
296  "mom":[0,0,0],
297  "e":[0,0,0],
298  "k":[4438.84,4438.84,4438.84],
299  "t":[0,0.258763,1],
300  "x":[-1.99299,65.9771,23],
301  "y":[-8.12482,-45.244,-70],
302  "z":[-225,-229.486,-233]
303 }
304 ]
305 ~~~
306 
307 where things inside of a pair of {} is the information of an event, "n" is the number of [Track Points](#track-point) in each event.
308 
309 ### ROOT
310 
311 ROOT use TTree to save data. TTree have entry and branch to build their data structure. Each branch is a variable listed in [Track Points](#track-point), and each entry is a event. ROOT will compress the data, so it will cost less disk space. ROOT can load part of data, which can save time when it get large.
